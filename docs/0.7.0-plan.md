# 0.7.0 — Remote Connectivity Execution Plan

**Status:** Plan approved. Phases 1, 2, 2b, 3a, 3b, 3c, 3d.1,
3e (rule-only), and 4b landed (offline-tested) on
`claude/add-camera-remote-connectivity-sJPbe`. Phase 3d.2
(real hole-punch wiring) and the CameraSession integration of
the 3e decision rule are deferred until a real Reolink camera
+ tcpdump are available — see "What's gated on real-device
data" below.
**Owner:** Eric.
**Last revised:** 2026-05-15.

This doc is the execution playbook for the rest of 0.7.0 — what
gets built, in what order, with what trade-offs locked in.
The architecture rationale lives in
[`docs/remote-connectivity.md`](./remote-connectivity.md); the
phase tasks below assume you've read that first.

## Release framing

0.7.0 ships **remote access only**. No bundled HKSV, no other
storylines — single-feature release for a clean validation
surface and a clean CHANGELOG. If Phase 4 slips past ~3 weeks,
defer the whole remote storyline to 0.8.0 rather than bundling
half-built work.

## Decisions

The 14 design questions that were open at planning time, with
the chosen answer and the path-not-taken implication noted so
this doc serves as a "why does it look like this" reference
later.

| # | Decision | Choice | What we're giving up |
|---|----------|--------|----------------------|
| 1 | Transport API shape | **Message-level (`BcMessage`)** | Less reusable in the abstract; coupled to ReolinkBaichuan |
| 2 | LAN/remote choice | **Lazy fallback, 3 s probe window** | First-frame latency off-LAN; users on cellular wait 3 s |
| 3 | Re-acquire LAN mid-session | **Stay on remote until session ends** | Reolink server cost when user comes home mid-stream |
| 4 | Relay opt-out | **None** | Power users can't refuse Reolink TURN; relay is rare anyway |
| 5 | Video pipeline abstraction | **Common `VideoSource` protocol** | Larger refactor up front (touches existing RTSP path) |
| 6 | Audio path | **Tee through existing AVAudioEngine** | Risk of RTSP-engine state leaking into remote mode |
| 7 | Recordings over remote | **Full parity via Baichuan file-download (msg_id=89)** | +5 d effort; reverse-engineering surface widens |
| 8 | Mode indicator | **Pip on each camera tile** | Slightly more grid visual noise |
| 9 | First-time disclosure | **Silent — mode indicator only** | Users won't know what "Remote" means without exploring |
| 10 | Opt-out grain | **Single app-wide toggle** | Users with mixed-trust cameras can't opt some out |
| 11 | Battery cameras | **Best-effort + "sleeping" badge from discovery server** | Aggressive wake-hint behavior of official app not matched |
| 12 | 0.7.0 scope | **Remote only** | HKSV and other queued storylines wait for 0.7.1 / 0.8.0 |
| 13 | Test matrix | **Single residential NAT + cellular** | Symmetric-NAT / CGNAT users discover bugs first |
| 14 | Server pool source | **Swift constants** | Reolink server rotation requires app update, not config push |

## Remaining work

Phase numbering continues from
[`docs/remote-connectivity.md`](./remote-connectivity.md). The
phases are not strictly sequential — 3a/3b/3c are mechanical
prep that can land in parallel with 4a; the integration points
(3d, 4c) are the bottlenecks.

### Phase 3 — Transport refactor + NAT traversal (~10 days)

Splits into sub-phases so each lands as a reviewable PR.

**Phase 3a — `BcMessageTransport` protocol (½ d).** Add to
`Sources/ReolinkBaichuan/` (peer with `BcMessage.swift`, not in
`ReolinkP2P` — the protocol's natural home is the module that
owns `BcMessage`). One method shape:

```swift
public protocol BcMessageTransport: Sendable {
    func connect() async throws
    func close() async
    func send(_ message: BcMessage) async throws
    func subscribe() -> AsyncStream<BcMessage>
    func sendAndAwait(_ message: BcMessage, timeout: TimeInterval, stage: String) async throws -> BcMessage
    func nextMessageNumber() async -> UInt16
    func currentCipher() async -> BcCipher
    func setCipher(_: BcCipher) async
}
```

This matches the surface `BaichuanClient` already exposes — the
protocol is just "factor the actor public API into a protocol".

**Phase 3b — `LANTransport` conformance (1 d).** New file
`Sources/ReolinkBaichuan/Transport/LANTransport.swift`. Wraps an
`NWConnection` (TCP/9000) and conforms to `BcMessageTransport`.
This is essentially `BaichuanClient`'s current internals lifted
into a separate type. Existing `BaichuanClient` stays unchanged
in 3b — the new type is additive.

**Phase 3c — `BaichuanClient` refactor (2 d, risk).** Hollow
out `BaichuanClient`: it becomes a thin actor that holds an
`any BcMessageTransport` and delegates the transport calls.
The login state machine (`BaichuanLogin`), encryption negotiation,
extension XML handling stay in `BaichuanClient` — only the
NWConnection-owning bits move out. **Critical:** every existing
behaviour (TCP reconnect, message-num counter rolling, etc.) has
to keep working. Mitigation: snapshot the current
`BaichuanTests` suite at HEAD, do the refactor, suite must stay
green before merge.

**Phase 3d — `RemoteTransport` (5 d, high risk).** New file
`Sources/ReolinkP2P/RemoteTransport.swift`. Conforms to
`BcMessageTransport` but routes via BcUdp data packets over a
hole-punched UDP `NWConnection`. Owns:

- A persistent BcUdp control channel to the discovered peer
  (built from the `P2PDiscovery` candidates)
- The hole-punch state machine: send Disc probes to all
  candidates, pick the first 5-tuple that completes a
  round-trip handshake, fall back to relay candidate after 6 s
- The Data-packet reassembly buffer (large BcMessages span
  multiple packets — split by sequence, ack, reassemble)
- The keepalive loop (ping every ~10 s so the NAT mapping
  doesn't expire)

**Phase 3e — Reachability state machine (1½ d).** Wire into
`CameraSession`. On `connect()`:

1. Attempt LAN reachability (the existing CGI probe — already
   in the connect path). If reachable within 3 s, build a
   `LANTransport` and proceed.
2. If LAN times out (camera unreachable on the local subnet),
   load `entry.uid`, build a `RemoteTransport`, proceed.
3. If the UID is missing, surface "Camera offline" (the user has
   never successfully connected on LAN, so we have no key for
   discovery).

Once remote is active, no LAN re-acquisition for the session
lifetime (decision #3). On reconnect (TCP drop, app foreground),
re-run the decision tree.

### Phase 4 — Video + audio pipeline (~12 days)

The single biggest chunk of remaining work. Splits into:

**Phase 4a — `VideoSource` protocol (1 d).** New file
`Sources/ReolinkStreaming/VideoSource.swift`. Common surface for
"a thing that produces decoded video frames + audio samples for
playback".

```swift
public protocol VideoSource: Sendable {
    func start() async throws
    func stop() async
    var frames: AsyncStream<DecodedFrame> { get }
    var audio: AsyncStream<AudioSample> { get }
}
```

**Phase 4b — `RTSPVideoSource` (RTSP path conforms; 2 d, risk).**
Refactor the existing RTSP playback to conform. Touches the
playback engine — needs careful regression testing. Existing
playback behaviour must be byte-identical (latency, codec
selection, snapshot semantics). The decode pipeline
(`VTDecompressionSession`) stays in place — only the source-of-
frames gets renamed.

**Phase 4c — `BaichuanVideoSource` (4 d).** New file
`Sources/ReolinkStreaming/BaichuanVideoSource.swift`. Consumes
`msg_id=3` frames from a `BcMessageTransport` (any flavor —
LAN or remote — though in practice it's only used over remote,
since LAN keeps using RTSP). Splits each message body into
Annex-B NALUs, feeds VideoToolbox. Per-frame metadata (PTS, key
frame flag) extracted from the Baichuan video sub-header.

**Phase 4d — Audio (1 d).** Baichuan AAC frames tee through the
existing AVAudioEngine sink (decision #6). One concern: the
RTSP path expects RTP timestamps; Baichuan uses its own clock.
Add a per-source clock-mapper so the audio engine sees a
monotonic time-base regardless of source.

**Phase 4e — Snapshot + talkback over remote (2 d).** Snapshot
(`msg_id=109`) is already in `BaichuanClient` for free since the
control-plane works over remote. Talkback (`msg_id=201/202`) is
in `BaichuanTalkback.swift` — should also work for free if
Phase 3c is clean.

**Phase 4f — Recordings parity (2 d).** Per decision #7: fetch
recordings over remote via Baichuan file-download. This is the
single largest reverse-engineering surface left in 0.7.0 because
the file-download protocol over Baichuan is less well-documented
than the live stream path. Risk mitigation: ship recordings-
remote behind an internal flag if the protocol turns out to be
brittle; fall back to "recordings work only on LAN" (the
originally-recommended option from question 7) if the flag
proves unstable in the TestFlight cohort.

**Phase 4g — `RecordingSource` plumbing (1 d).** Mirror Phase
4a/b for recordings: a small `RecordingSource` abstraction so
the player picks between RTSP-over-HTTP (LAN) and Baichuan
file-download (remote) transparently.

### Phase 5 — UI (~3 days)

**Phase 5a — Connection-mode pip on tiles (1 d).** Small
colored dot in the bottom-left of each camera tile. Three
states:

- **Green pip**: LAN
- **Amber pip**: Remote (direct P2P)
- **Orange pip**: Remote (relayed through Reolink)

Tap the pip → small popover with the three-state explainer
text. Per decision #8.

**Phase 5b — "Sleeping" badge for battery cams (½ d).** When
discovery reports the camera as sleeping (per decision #11), the
tile shows a small `moon.zzz` icon overlaid on the last-known
preview. Tile is tappable but live stream stays gated until the
discovery server reports the camera awake. Diagnostic-only — we
don't try to wake the camera ourselves (would need reverse-
engineered FCM-style push that's out of scope).

**Phase 5c — Settings opt-out (½ d).** Settings → Privacy & Sync
→ new "Remote access" section. Single toggle, "Allow remote
access", default ON. When OFF, the reachability state machine
in Phase 3e skips the remote-fallback step and shows "Camera
offline" instead. Per decisions #4 / #10.

**Phase 5d — Privacy-stance copy (½ d).** Edit `README.md` and
the in-app About screen per the agreed framing:

> At home, Reolens talks only to your cameras on your local
> network. Away from home, Reolens uses Reolink's discovery
> servers (and, if direct hole-punching fails, their relay
> servers) to reach your camera. There is still no Reolens
> server. You can disable remote access in Settings if you
> only want LAN-mode behavior.

**Phase 5e — Diagnostics integration (½ d).** Surface the
discovery `.exhausted` error in Diagnostics Center: per-server
attempt list with timed-out / unreachable / malformed / empty
counts. Helps support threads when "remote isn't working" turns
out to be a regional server outage.

**Skipped:** first-time disclosure banner (decision #9 — silent
by design).

### Phase 6 — Release (~1 day)

- Tag, CHANGELOG finalize, docs sweep.
- Per-NAT validation (decision #13): single residential NAT +
  cellular. CGNAT / symmetric-NAT users discover any remaining
  bugs in the TestFlight cohort.
- Beta soak ~1 week before promoting.

## Risk register

### High risk

- **Phase 3c — `BaichuanClient` refactor.** This touches the
  most heavily-exercised file in the codebase. Mitigation: the
  existing `BaichuanTests` + `BaichuanWireTests` suites must
  stay green before merge; no behavioural change is acceptable.
- **Phase 3d — NAT traversal.** Real-world NAT behaviour
  varies wildly; the offline tests can't simulate this.
  Mitigation: ship behind a feature flag, soak in TestFlight for
  a week before promoting.
- **Phase 4c — Baichuan video pipeline.** Largest LOC delta of
  the project; codec edge cases (especially on battery cameras)
  are hard to predict. Mitigation: start with H.264 main-stream
  only, add H.265 + sub-stream support iteratively once the
  main path is stable.
- **Phase 4f — Recordings over Baichuan file-download.** Least-
  documented protocol surface. Mitigation: internal flag to
  fall back to "recordings LAN-only" if the protocol misbehaves.

### Medium risk

- **Phase 4b — RTSP `VideoSource` refactor.** Touches working
  code. The original RTSP path is fine today; the risk is
  introducing a regression while abstracting it. Mitigation:
  byte-identical latency / first-frame target before merge.
- **Audio clock-mapping in Phase 4d.** Baichuan and RTP have
  different clock bases; if the mapper drifts, audio falls out
  of sync after ~30 s. Mitigation: re-sync on every key frame.

### Low risk

- Everything in Phase 5. UI work is well-trodden in this
  codebase.
- Phase 6 release mechanics.

## What's gated on real-device data

The structural plumbing of Phase 3 is in place. What remains
is the wire-level work that only a tcpdump capture against a
real `p2p*.reolink.com` exchange (and a real camera on a real
NAT) can validate. Specifically:

- **Phase 3d.2** — replace
  `RemoteTransport.sendAndAwait`'s
  `RemoteTransportError.notYetImplemented` with the actual
  hole-punch state machine, multi-packet reassembly via
  BcUdp Data/Ack, the keepalive loop, and the
  `NWConnectionBcUdpDataConnection` concrete that owns the
  UDP socket.
- **CameraSession integration of `CameraReachability`** —
  the decision rule is committed and unit-tested (Phase 3e
  in `Sources/AppShared/CameraReachability.swift`), but the
  `startBaichuanEvents()` site still always builds a
  `LANTransport`. Wiring the rule in today would change
  observable behaviour to "camera tile loads but has no
  video" off-LAN; deferred to land with Phase 4 once the
  video pipeline can actually carry frames over the remote
  transport.

## Real-device findings (2026-05-16 pcap)

An 88 MB / 74k-packet capture from the Reolink macOS app
connecting to a camera off-LAN against `p2p*.reolink.com`
resolved several open items and uncovered one new one:

### Resolved

- **BcUdp magic constants** — all three Phase 1 values were
  wrong (different discriminator byte; Data ↔ Ack were also
  swapped). Wire-truth values landed in
  `Sources/ReolinkBcUdp/Wire/BcUdpConstants.swift`:
  - Disc: `0x2A87CF3A` (wire LE `3A CF 87 2A`)
  - Data: `0x2A87CF10` (wire LE `10 CF 87 2A`)
  - Ack:  `0x2A87CF20` (wire LE `20 CF 87 2A`)
- **Byte order** — the wire is little-endian throughout.
  Phase 1 used big-endian helpers; all helpers + encoders +
  decoders flipped to LE.
- **Header sizes** — Disc and Data are each 20 bytes
  (5 × u32 fields), Ack is a fixed 28 bytes. Was 12 / 22 / 18
  in Phase 1; codec corrected.
- **Field layouts** — each kind's header fields re-derived
  from the capture. Notably Data carries `(connectionID,
  reserved=0, sequence, payloadSize)` after the magic — no
  separate `unknown_a` field as neolink labels it. The
  per-fragment `payloadSize` is u32 (not u16), so a single
  fragment can carry up to 4 GB if Reolink ever needs it.
- **Inner Baichuan framing** — Data packets carry standard
  Baichuan TCP-framed messages with magic `0xABCDEF0` at
  offset 20 of the UDP payload. The existing
  `ReolinkBaichuan.BcMessage.decode(from:cipher:)` works
  as-is once the receiver has assembled all fragments by
  sequence number. No new Baichuan codec is needed for the
  remote path.

### Resolved (full state machine, 2026-05-16, offline)

- **DataReassembler / DataFragmenter** — pure value types
  for inbound fragment ordering and outbound MTU-sized
  splitting. Cover out-of-order delivery, duplicates, gap
  fills, and round-trip across each other.
- **HolePunchScheduler** — direct-first / relay-fallback
  state machine with injected probe runner. Returns
  `.direct` or `.relayed` for the UI.
- **RemoteTransport receive loop + sendAndAwait** — wires
  the reassembler/fragmenter into the actor, drives the
  hole-punch via scheduler, dispatches inbound BcMessages
  to reply slots or subscribers. Race-free sequencing
  preserved from LANTransport.
- **NWConnectionBcUdpDataConnection** — concrete UDP
  channel that wraps an already-open `NWConnection.udp`
  and runs a receive loop. Loopback-tested via POSIX
  socket.
- **NWConnectionBcUdpPunchEngine** — concrete
  HolePunchProbeRunner that opens a UDP socket per
  candidate, sends a Disc probe, caches the connection on
  success so the data plane reuses the same 5-tuple
  (preserves the NAT mapping).
- **`RemoteTransport.production(...)` factory** wires it
  all together against the public `p2p*.reolink.com`
  cluster.

### Resolved (cipher port, 2026-05-16)

- **Disc payload obfuscation** — solved. The wire cipher
  is an XOR stream against a 32-byte keystream derived
  from 8 hard-coded u32 constants plus the BcUdp Disc
  header's `senderID` field as the offset. Ported from
  neolink's `crates/core/src/bcudp/xml_crypto.rs` into
  `Sources/ReolinkBcUdp/Wire/DiscoveryXMLCrypto.swift`.
  Validated end-to-end against the captured success
  reply: 348 bytes of wire data → BcUdpPacket.decode →
  DiscoveryXMLCrypto.decrypt → DiscoveryXML.LookupResponse
  → `registration = 172.232.163.180:58200`,
  `relay = 172.232.163.180:58101`.
- **Discovery XML schema** — the Phase 1 schema was wrong
  on every wrapper and field name. Real wire format:
  request is `<P2P><C2M_Q><uid/><ver/><family/><s/></C2M_Q></P2P>`,
  reply is `<P2P><M2C_Q_R><reg><ip/><port/></reg>` +
  `<relay><ip/><port/></relay>` + status fields. Schema
  rewritten in `DiscoveryXML.swift` to match wire.
- **P2PDiscovery wire integration** — the actor now
  encrypts request payloads with the request `senderID`
  before sending, and decrypts reply payloads with the
  reply `senderID` (which the server stamps into its own
  Disc header). 440 tests pass including an end-to-end
  pin against captured wire bytes.

## Open items needing real-device data

Things this plan can't resolve from a sandbox / cold-write
pass — they need a compiler + a Reolink camera:

1. **BcUdp magic constants** in
   `Sources/ReolinkBcUdp/Wire/BcUdpConstants.swift`. Round-trip
   tests pass with whatever values are set; only a `tcpdump`
   capture against a real `p2p*.reolink.com` confirms them.
2. **Discovery XML tag names** in
   `Sources/ReolinkBcUdp/Wire/DiscoveryXML.swift`'s `Tag` enum.
   Same risk profile as the magics — single-place edit if wrong.
3. **`p2p*.reolink.com` server-pool composition** in
   `Sources/ReolinkP2P/DiscoveryServerPool.swift`. Likely
   accurate from public references, but Reolink can rotate.
4. **Baichuan video sub-header layout.** Required for Phase 4c
   NALU extraction. Documented in `neolink/crates/core/src/bc/`
   but not yet validated in this project.
5. **Baichuan file-download msg_id and payload schema.**
   Required for Phase 4f. Needs reverse-engineering surface.

## What's NOT in 0.7.0

Explicit non-goals so we don't scope-creep:

- **Multi-device call sync.** When two of the user's Apple
  devices simultaneously open the same remote-mode camera, they
  each open their own P2P channel — there's no sharing. The
  official Reolink app behaves the same way; documenting for
  clarity.
- **Camera-side P2P configuration UI.** Reolens never has to
  configure the camera's own P2P settings — the camera handles
  its own registration with Reolink's servers. If the user has
  P2P disabled on the camera, remote access just won't work and
  they'll see "Camera offline" away from home. Out of scope for
  Reolens to detect and explain this.
- **iCloud sync of the app-wide "Allow remote access" toggle.**
  Device-local preference. Different devices may have different
  network situations (e.g. a Mac that's always at home vs. an
  iPhone that's away half the time).
- **TestFlight-side telemetry for connection-mode usage.**
  Reolens has no analytics by design; we're not adding any
  for 0.7.0. Support diagnostics surface in Phase 5e is the
  only signal we get.

## Sequencing recommendation

Order the work to make the riskiest sub-phases reach a real
device first:

1. Phase 3a, 3b, 3c (transport refactor — mechanical, but
   gates everything else).
2. Phase 3e in parallel with 3d's first half (the reachability
   state machine can be unit-tested with a stubbed transport,
   landing it early de-risks the integration).
3. Phase 4a, 4b (the RTSP `VideoSource` refactor — risk to
   existing playback, do it while the test suite is fresh in
   mind).
4. Phase 3d second half + Phase 4c (the actual remote
   connection becomes end-to-end testable here — first time we
   can stream a real camera over real internet).
5. Phase 4d-g sequentially.
6. Phase 5 in parallel with the back half of Phase 4 —
   independent code paths.
7. Phase 6.

Estimated wall-clock with one developer + the compiler +
device available: ~6-7 weeks elapsed, depending on how much
the recordings-parity protocol fights back.
